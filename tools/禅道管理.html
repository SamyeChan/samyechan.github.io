# Day 01

目标：理解禅道管理流程的模型以及实际操作

# 禅道工具研发的背景、解决项目遇到的以下痛点

  • 需求频繁变更
  • 多次回归测试，Bug不能收敛
  • 交付周期长
  • 团队很努力，但无效率
  • 承诺期限，但出现延期
  • 人员流动大
  • 交付不被市场/客户接受

# 禅道软件概述

  • 禅道软件的特点
    1. 全生命周期管理
    2. 支持敏捷项目管理
    3. 易于扩展
  • 禅道软件和同类软件比较
  • 不同角色的操作简介
    1. 管理者
    2. 产品经理
    3. 项目经理
    4. 开发人员
    5. 测试人员
    6. 运维和运营人员

# 针对产品开发，禅道的流程管理模型 —— Scrum

  • 典型 Scrum 管理框架
  • Scrum 团队
    1. 跨职能团队
    2. 团队的角色协作方式
  • 如何保证交付物和客户需求相符
    1. 建立需求列表（Product Backlog）
    2. 迭代增量交付
    3. 持续优先级管理
    4. 迭代验收，得到客户反馈
  • 如何提升交付物的质量和效率
    1. 好的计划会议是交付成功的基石
    • 迭代目标是解决交付风险第一步
    • 团队如何影响 Sprint Backlog
    • PO 如何影响 Sprint Backlog
    1. 透明化管理迭代过程
    • 燃尽图，发现迭代的风险
    • 白板和站立会议，让项目更加透明
    1. 和极限编程（XP）的结合
    • 结对编程
    • 持续集成
    1. 持续改进
    • 通过回顾得到改进方案

# 禅道软件实践操作演示：针对产品开发，禅道如何和 Scrum 结合

  • 案例场景：产品开发，要求两周交付一个可以上线的版本

    3. 设立部门

    2. 需求的创建、评审和变更管理
    3. 基于模块的需求拆分
    4. 需求的优先级管理

  • 项目经理视角：项目管理

    3. 确定项目需求
    4. 任务分解
    5. 项目进度透明化
    6. 项目验收
    7. 持续改进——项目总结会议
  • 开发人员视角：编码和修复缺陷

    2. 认领任务
    3. 创建版本
    4. 提交测试
    5. 修改BUg
    6. 文档管理
  • 测试人员视角：质量控制
    1. 创建测试任务
    2. 测试前准备——测试用例
    3. Bug提交和验证
  • 运维人员视角
    1. 提交产品改进需求
    2. 流模式工作：快速交付

  • 运营人员视角
